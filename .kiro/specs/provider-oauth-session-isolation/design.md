# Provider OAuth Session Isolation — Bugfix Design

## Overview

The provider OAuth flow for connecting Google Drive reuses the login callback (`/callback/google`), which unconditionally creates a Lucia session for whatever Google account completes the OAuth consent. If the user picks a different Google account than the one they're logged in with, their auth session is silently replaced — a confused-deputy session hijack. Additionally, the refresh token lands on `users.googleRefreshToken` instead of the provider record, and the frontend shows the wrong email.

The fix introduces a fully isolated provider OAuth flow: a dedicated initiation endpoint (`/providers/connect/google`), a dedicated callback (`/callback/provider/google`), CSRF-protected state carrying `userId` + flow type, credential storage directly on `user_providers`, and dropping `users.googleRefreshToken` entirely. The login flow is simplified to request only identity scopes. This is a breaking change — existing Google Drive providers must be re-connected.

## Glossary

- **Bug_Condition (C)**: A user initiates the provider OAuth flow and the shared `/callback/google` handler processes the return — creating or replacing a Lucia session instead of storing provider credentials
- **Property (P)**: The provider OAuth callback stores the refresh token in `user_providers.credentials` and the provider email in `user_providers.config.accountEmail`, then redirects without touching the session cookie
- **Preservation**: The login flow (`/login/google` → `/callback/google`) continues to create sessions exactly as before; Google Sheets sync and restore continue to work (reading from provider credentials instead of `users.googleRefreshToken`)
- **`oauthStateStore`**: In-memory `Map` in `auth/routes.ts` that stores OAuth state, code verifier, and metadata between initiation and callback
- **`pendingProviderCredentials`**: In-memory `Map` keyed by `userId:nonce` that temporarily stores provider OAuth credentials between the callback and provider creation. TTL of 10 minutes.
- **`nonce`**: A random string generated by the frontend before OAuth redirect, used to correlate the OAuth callback result with the correct browser tab/session. Stored in `sessionStorage` and passed through the OAuth flow.
- **`user_providers`**: Table storing provider connections per user — `credentials` (encrypted JSON), `config` (JSON with `accountEmail`, `rootPath`, etc.)
- **`users.googleRefreshToken`**: Legacy column that stored the Google refresh token on the user record — being dropped directly from the initial schema (breaking change)

## Bug Details

### Bug Condition

The bug manifests when a user clicks "Connect Google Account" on the provider form, which navigates to `/reauth/google`. This endpoint generates OAuth state and redirects to Google. When Google redirects back to `/callback/google`, the handler:
1. Exchanges the code for tokens
2. Looks up or creates a user record for the Google account
3. Creates a new Lucia session and sets the session cookie

This overwrites the user's existing session. If they picked a different Google account, they're now logged in as that account (or a brand-new user is created).

**Formal Specification:**
```
FUNCTION isBugCondition(request)
  INPUT: request of type HTTPRequest (the OAuth callback)
  OUTPUT: boolean

  state := oauthStateStore.get(request.query.state)
  RETURN state IS NOT NULL
         AND state.flowType IS UNDEFINED   -- old flow has no flowType marker
         AND state.returnTo IS NOT NULL     -- returnTo present = came from provider form
         AND callbackEndpoint = '/callback/google'  -- shared with login
END FUNCTION
```

### Examples

- User A (logged in as `alice@gmail.com`) connects Google Drive, picks `alice-work@gmail.com` during consent → session cookie now points to `alice-work@gmail.com` (or a new user), Alice is logged out
- User A connects Google Drive, picks the same `alice@gmail.com` → session is replaced with a new session for the same user (unnecessary session churn, refresh token stored on `users` not provider)
- User A connects Google Drive, picks a Google account with no existing user record → a new user record is created, session switches to that new user
- User A connects Google Drive, cancels OAuth consent → redirected back to `/callback/google` with error, which throws a 400 (no friendly redirect to the form)

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Login flow (`/login/google` → `/callback/google`) continues to create/find users and create Lucia sessions exactly as today
- Mouse clicks, form submissions, and all non-OAuth interactions on the provider form are unaffected
- Session refresh (`/auth/refresh`) and session validation (`/auth/me`) work identically
- Photo upload/download via `StorageProviderRegistry` works identically (registry already reads `credentials` from `user_providers`)
- Backup ZIP creation and restore-from-file flows are unaffected (they don't use Google tokens)

**Scope:**
All requests that do NOT hit the new `/providers/connect/google`, `/callback/provider/google`, or `/providers/pending/:nonce` endpoints are completely unaffected. The only behavioral change to existing code is:
- `google-sheets-service.ts` `getUserToken()` reads from `user_providers.credentials` instead of `users.googleRefreshToken`
- `restore.ts` `getUserWithToken()` reads from `user_providers.credentials` instead of `users.googleRefreshToken`
- `/callback/google` stops writing to `users.googleRefreshToken` (column is dropped)
- `/reauth/google` is removed (replaced by `/providers/connect/google`)
- `/login/google` scopes reduced to `openid`, `profile`, `email` only (no Drive scopes)

## Hypothesized Root Cause

Based on code analysis, the root causes are:

1. **Shared callback endpoint**: `/callback/google` handles both login and provider connection flows identically — it always creates a Lucia session and sets the session cookie. There is no `flowType` marker in the OAuth state to distinguish the two flows.

2. **No session preservation logic**: The callback unconditionally calls `lucia.createSession()` and `setCookie()`. There is no code path that skips session creation and instead stores credentials on a provider record.

3. **Credential storage on wrong entity**: The `/callback/google` handler stores the refresh token on `users.googleRefreshToken`. The provider form then sends `credentials: {}` (empty) to `POST /api/v1/providers`, so the provider record's `credentials` column contains encrypted empty JSON. The `StorageProviderRegistry.createProviderInstance()` then fails with "missing refreshToken" when it tries to use the provider.

4. **Missing provider email capture**: The callback fetches the Google user's email but only uses it for user record lookup/creation. It never stores the provider account's email in `user_providers.config.accountEmail`. The frontend falls back to `authStore.user.email`, which is the login account — not necessarily the provider account.

5. **No CSRF protection for provider flow**: The OAuth state stores `{ codeVerifier, createdAt, returnTo }` but not the authenticated user's ID. A malicious actor could potentially initiate an OAuth flow and have another user's callback complete it.

## Correctness Properties

Property 1: Bug Condition — Provider OAuth callback never modifies the session

_For any_ valid provider OAuth callback request to `/callback/provider/google` where the state is valid and the token exchange succeeds, the handler SHALL NOT call `lucia.createSession()`, SHALL NOT call `setCookie()` for the session cookie, and SHALL NOT insert into or update the `users` table. The response SHALL be a redirect to the frontend with `provider_connected=true` and `nonce` query parameters (no PII in URLs).

**Validates: Requirements 2.2, 2.3, 2.4**

Property 2: Preservation — Login OAuth callback behavior unchanged

_For any_ login OAuth callback request to `/callback/google` (the existing endpoint), the handler SHALL continue to create or find a user record, create a Lucia session, set the session cookie, and redirect to the dashboard (or `returnTo`). The behavior SHALL be identical to the pre-fix implementation, preserving all existing login functionality.

**Validates: Requirements 3.1, 3.2**

Property 3: Bug Condition — Provider credentials stored on provider record

_For any_ valid provider OAuth callback where the token exchange returns a refresh token, the handler SHALL store `{ refreshToken, email }` in `pendingProviderCredentials[userId:nonce]` (for new providers) or update the existing provider's `credentials` and `config.accountEmail` directly (for re-auth of existing providers via `providerId`).

**Validates: Requirements 2.2, 2.7**

Property 4: Preservation — Google Sheets sync and restore use provider credentials

_For any_ Google Sheets sync or restore-from-Drive operation, the service SHALL read the refresh token from the Google Drive provider's `user_providers.credentials` column (not from `users.googleRefreshToken`), and SHALL produce the same sync/restore behavior as before.

**Validates: Requirements 3.4**

Property 5: Security — OAuth state validates session identity

_For any_ provider OAuth callback, the handler SHALL verify that the current session's `userId` matches the `userId` stored in the OAuth state. If they do not match, the handler SHALL reject the request and redirect with an error.

**Validates: Requirements 5.1, 5.2**

Property 6: Security — No PII in redirect URLs

_For any_ provider OAuth callback redirect, the URL SHALL contain only opaque tokens (`nonce`) and status flags (`provider_connected`, `provider_error`) — never email addresses or other PII. The frontend retrieves PII via authenticated API calls using the nonce.

**Validates: Security best practice (OWASP)**


## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct:

**File**: `backend/src/api/auth/routes.ts`

**New endpoint**: `GET /providers/connect/google`
1. **Require authentication**: Validate the session cookie, reject if not authenticated
2. **Accept parameters**: `returnTo` (required), `nonce` (required — generated by frontend), `providerId` (optional, for re-auth of existing provider), `email` (optional, for `login_hint`)
3. **Generate OAuth state**: Store `{ codeVerifier, createdAt, returnTo, nonce, userId, flowType: 'provider', providerId }` in `oauthStateStore`
4. **Build authorization URL**: Use scopes `openid`, `profile`, `email`, `https://www.googleapis.com/auth/drive.file`; set `access_type=offline`, `prompt=consent`; optionally set `login_hint` from `email` param
5. **Redirect to Google**

**New endpoint**: `GET /callback/provider/google`
1. **Validate state**: Look up state in `oauthStateStore`, verify `flowType === 'provider'`
2. **Validate session**: Read session cookie, verify `session.userId === state.userId` (CSRF protection)
3. **Exchange code for tokens**: Use `google.validateAuthorizationCode()`
4. **Fetch Google user info**: Get email from the Google account
5. **Handle missing refresh token**: If `tokens.hasRefreshToken()` is false, redirect with `?provider_error=no_refresh_token`
6. **Store credentials**:
   - If `state.providerId` exists (re-auth of existing provider): update that provider's `credentials` with `{ refreshToken }` (encrypted) and set `config.accountEmail` to the Google account's email
   - If no `providerId` (new provider): store `{ refreshToken, email }` in `pendingProviderCredentials[userId:nonce]` with 10-minute TTL
7. **Redirect to frontend**: `returnTo` + `?provider_connected=true&nonce=<nonce>` — no PII in URLs
8. **Error handling**: On any failure, redirect to `returnTo` + `?provider_error=<reason>`

**New endpoint**: `GET /providers/pending/:nonce`
1. **Require authentication**: Validate session cookie
2. **Look up pending credentials**: Find `pendingProviderCredentials[userId:nonce]`
3. **Return email only**: `{ success: true, data: { email } }` — does NOT return the refresh token, does NOT consume the entry
4. **404 if not found or expired**
5. **Route location**: This endpoint lives in `providers/routes.ts` (mounted at `/api/v1/providers`), not in `auth/routes.ts`

**Endpoint removal**: `GET /reauth/google`
- Remove this endpoint entirely — its only purpose was provider connection, which is now handled by `/providers/connect/google`

**File**: `backend/src/api/auth/routes.ts` — existing `/login/google`
1. **Reduce scopes**: Request only `openid`, `profile`, `email` — remove `https://www.googleapis.com/auth/drive.file`
2. **Change prompt**: Use `prompt=select_account` instead of `prompt=consent` — users see the account picker but don't re-consent to scopes they've already granted
3. **Remove `access_type=offline`**: Login no longer needs a refresh token since it doesn't access Drive — session-based auth via Lucia is sufficient

**File**: `backend/src/api/auth/routes.ts` — existing `/callback/google`
1. **Stop writing `googleRefreshToken`**: Remove the `googleRefreshToken` field from the `INSERT` and `UPDATE` statements in the login callback
2. **No other changes**: Session creation, user lookup/creation, and redirect logic remain identical

**File**: `backend/src/api/auth/lucia.ts`

2. **Remove from `DatabaseUserAttributes` interface**: Remove `googleRefreshToken: string | null`
3. **Remove from Lucia `User` interface declaration**: Remove `googleRefreshToken: string | null`

**File**: `backend/src/db/schema.ts`

10. **Update `canSave` logic**: For Google Drive, change from `isEditMode || !!authStore.user?.email` to `isEditMode || !!pendingEmail` (where `pendingEmail` is the email retrieved from the pending credentials API)

**File**: `frontend/src/lib/components/settings/GoogleDriveProviderForm.svelte`

**File**: `frontend/src/lib/components/settings/AddProviderDialog.svelte`
- No changes needed — this component shows provider type selection and a "Connect your Google account first" message. It does not trigger OAuth directly.

**Google Drive config shape**: `{ rootPath: string, accountEmail?: string }`. The `accountEmail` field is set by the provider OAuth callback and displayed by the frontend. It is optional because S3 providers don't have it.

### Pending Provider Credentials Store

A dedicated in-memory `Map` for temporarily storing provider OAuth credentials between the callback and provider creation:

```typescript
const pendingProviderCredentials = new Map<string, {
  refreshToken: string;
  email: string;
  createdAt: number;
}>();
```

**Key format**: `userId:nonce` — the nonce is generated by the frontend and passed through the entire OAuth flow, ensuring each browser tab has its own credential slot.

**Module location**: `backend/src/utils/pending-credentials.ts` — a shared module exporting the `pendingProviderCredentials` Map and helper functions (`storePending`, `getPendingEmail`, `consumePending`, `cleanupExpired`). Both `auth/routes.ts` (callback writes) and `providers/routes.ts` (create consumes) import from this module.

**Max size**: Capped at 1000 entries. When the limit is reached, the oldest entry is evicted before inserting a new one. This prevents unbounded memory growth per project conventions.

**Lifecycle**:
1. Created by `/callback/provider/google` after successful token exchange
2. Read (email only) by `GET /providers/pending/:nonce`
3. Consumed and deleted by `POST /api/v1/providers` when creating a Google Drive provider
4. Expired entries cleaned up on access (10-minute TTL, same as OAuth state)

**Edge cases**:
- Multi-tab: Each tab generates its own nonce, so credentials don't collide. If a user completes OAuth in tab 2, tab 1's nonce won't match any stored credentials — save will fail with "Please connect your Google account first"
- Server restart: Pending credentials are lost. User re-does the OAuth flow. Acceptable for a self-hosted app.
- Nonce reuse: Nonces are UUIDs — collision probability is negligible

### OAuth State Store Type Update

The `oauthStateStore` type is updated to support both login and provider flows:

```typescript
const oauthStateStore = new Map<string, {
  codeVerifier: string;
  createdAt: number;
  returnTo?: string;
  // Provider flow fields (absent for login flow)
  userId?: string;
  flowType?: 'provider';
  providerId?: string;
  nonce?: string;
}>();
```

### Re-authentication Flow (existing provider)

When editing an existing provider and clicking "Change":
1. Frontend passes `providerId` and the current `config.accountEmail` as `email` hint to `/providers/connect/google`
2. Google pre-selects the account via `login_hint`
3. Callback updates the existing provider's `credentials` and `config.accountEmail` directly (no pending store needed)
4. Redirect back to the edit page — form reloads from server, showing updated email
5. No form state is lost because the callback only touches credentials/email on the server

### Provider Re-connection UX (breaking change)

After the schema change, existing Google Drive providers with empty credentials will fail when used. The provider list/card should detect this:
- When `StorageProviderRegistry.createProviderInstance()` throws "missing refreshToken", the provider card shows a "Re-connect" prompt
- The edit page shows the "Connect Google Account" button instead of the connected state when `config.accountEmail` is missing

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code, then verify the fix works correctly and preserves existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm or refute the root cause analysis.

**Test Cases**:
1. **Session replacement test**: Initiate `/reauth/google` as User A, mock Google returning a different account's tokens → verify the session cookie changes (demonstrates the bug)
2. **New user creation test**: Initiate `/reauth/google` as User A, mock Google returning an unknown account → verify a new user record is created (demonstrates the bug)
3. **Empty credentials test**: Complete the provider OAuth flow, then check `user_providers.credentials` → verify it contains encrypted `{}` (demonstrates the credential storage bug)
4. **No flow distinction test**: Inspect the OAuth state stored by `/reauth/google` → verify it has no `flowType` field (confirms root cause)

### Fix Checking

**Pseudocode:**
```
FOR ALL request WHERE isBugCondition(request) DO
  sessionBefore := getSessionCookie(request)
  result := handleProviderCallback(request)
  sessionAfter := getSessionCookie(result)

  ASSERT sessionBefore = sessionAfter                    -- session unchanged
  ASSERT result.status = 302                             -- redirect
  ASSERT result.location CONTAINS 'provider_connected=true'
  ASSERT result.location DOES NOT CONTAIN '@'            -- no PII in URL
  ASSERT pendingProviderCredentials HAS entry for userId:nonce
  ASSERT users TABLE row count unchanged
END FOR
```

### Preservation Checking

**Pseudocode:**
```
FOR ALL request WHERE NOT isBugCondition(request) DO
  ASSERT handleLoginCallback_fixed(request) = handleLoginCallback_original(request)
  ASSERT lucia.createSession() IS called
  ASSERT setCookie() IS called with session cookie
  ASSERT user record IS created or found
END FOR
```
