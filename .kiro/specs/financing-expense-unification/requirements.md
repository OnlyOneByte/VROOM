# Requirements Document

## Introduction

This document defines the requirements for unifying financing payments with the expense system in VROOM. Currently, vehicle financing payments are tracked in a separate `vehicle_financing_payments` table, disconnected from the expense system. This feature eliminates that table and makes financing payments a special type of expense identified by an `isFinancingPayment` boolean flag. The backend automatically adjusts the vehicle's financing balance on expense create, update, and delete operations. The financing tab derives payment history, principal/interest breakdowns, and amortization data from expenses combined with existing calculation utilities.

## Glossary

- **Expense**: A record in the `expenses` table representing any vehicle cost, including financing payments.
- **Financing_Payment**: An expense with `isFinancingPayment === true` and `category === 'financial'` that represents a loan or lease payment.
- **Vehicle_Financing**: A record in the `vehicle_financing` table containing loan or lease configuration (original amount, APR, term, payment schedule) and the current balance.
- **Current_Balance**: The `currentBalance` field on a Vehicle_Financing record, representing the remaining amount owed.
- **Original_Amount**: The `originalAmount` field on a Vehicle_Financing record, representing the initial loan or lease amount.
- **Financing_Hook**: Backend logic that intercepts expense CRUD operations and adjusts the Vehicle_Financing balance when the expense is a Financing_Payment.
- **Derived_Payment_Entry**: A frontend data structure computed from a Financing_Payment expense combined with Vehicle_Financing configuration, containing payment number, remaining balance, principal/interest breakdown, and payment type.
- **Amortization_Schedule**: A precomputed table of principal and interest amounts per payment period, generated by `financing-calculations.ts`.
- **Expense_Form**: The frontend form component used to create and edit expenses.
- **Financing_Tab**: The tab on the vehicle detail page that displays financing information and payment history.
- **Expense_API**: The backend REST API endpoints for expense CRUD operations (`/api/v1/expenses`).
- **Balance_Delta**: The difference between old and new expense amounts used to adjust the financing balance during updates.

## Requirements

### Requirement 1: Expense Schema Extension

**User Story:** As a developer, I want the expenses table to include an `isFinancingPayment` boolean column, so that financing payments can be distinguished from regular expenses without relying on tags or conventions.

#### Acceptance Criteria

1. THE Expense schema SHALL include an `isFinancingPayment` column of type boolean with a default value of `false`
2. WHEN an expense is created without specifying `isFinancingPayment`, THE Expense_API SHALL default the value to `false`
3. WHEN an expense is queried, THE Expense_API SHALL include the `isFinancingPayment` field in the response

### Requirement 2: Drop Vehicle Financing Payments Table

**User Story:** As a developer, I want the `vehicle_financing_payments` table removed from the schema, so that the codebase has a single source of truth for all vehicle costs.

#### Acceptance Criteria

1. THE database schema SHALL NOT contain a `vehicle_financing_payments` table
2. THE codebase SHALL NOT contain type definitions, repository methods, or route handlers referencing `vehicle_financing_payments`
3. THE codebase SHALL NOT contain the `FINANCING_TAGS` constant or tag-based financing payment detection logic

### Requirement 3: Financing Balance Adjustment on Expense Creation

**User Story:** As a user, I want the financing balance to decrease automatically when I record a financing payment as an expense, so that I do not have to manually update the balance.

#### Acceptance Criteria

1. WHEN an expense with `isFinancingPayment === true` is created AND the vehicle has active financing, THE Financing_Hook SHALL subtract the `expenseAmount` from the Vehicle_Financing `currentBalance`
2. WHEN the subtraction would result in a negative balance, THE Financing_Hook SHALL clamp the `currentBalance` to zero
3. WHEN the `currentBalance` reaches zero (within a tolerance of 0.01), THE Financing_Hook SHALL mark the Vehicle_Financing as completed by setting `isActive` to `false` and `endDate` to the expense date
4. WHEN an expense with `isFinancingPayment === true` is created AND the vehicle has no active financing, THE Expense_API SHALL reject the request with a validation error
5. WHEN an expense with `isFinancingPayment === false` is created, THE Financing_Hook SHALL NOT modify any Vehicle_Financing record

### Requirement 4: Financing Balance Reversal on Expense Deletion

**User Story:** As a user, I want the financing balance to be restored when I delete a financing payment expense, so that accidental deletions do not permanently corrupt the balance.

#### Acceptance Criteria

1. WHEN an expense with `isFinancingPayment === true` is deleted, THE Financing_Hook SHALL add the `expenseAmount` back to the Vehicle_Financing `currentBalance`
2. WHEN the addition would exceed the `originalAmount`, THE Financing_Hook SHALL clamp the `currentBalance` to the `originalAmount`
3. WHEN a previously auto-completed Vehicle_Financing has its balance restored above 0.01, THE Financing_Hook SHALL reactivate the financing by setting `isActive` to `true` and `endDate` to null
4. WHEN an expense with `isFinancingPayment === false` is deleted, THE Financing_Hook SHALL NOT modify any Vehicle_Financing record

### Requirement 5: Financing Balance Delta on Expense Update

**User Story:** As a user, I want the financing balance to adjust correctly when I edit a financing payment expense, so that amount changes and flag toggles are reflected accurately.

#### Acceptance Criteria

1. WHEN a Financing_Payment expense amount is changed from `oldAmount` to `newAmount`, THE Financing_Hook SHALL adjust the `currentBalance` by the Balance_Delta (`oldAmount - newAmount`)
2. WHEN an expense's `isFinancingPayment` flag changes from `true` to `false`, THE Financing_Hook SHALL reverse the original deduction by adding back the `oldAmount` to the `currentBalance`
3. WHEN an expense's `isFinancingPayment` flag changes from `false` to `true`, THE Financing_Hook SHALL apply a new deduction by subtracting the `newAmount` from the `currentBalance`
4. WHEN an expense that was not and remains not a Financing_Payment is updated, THE Financing_Hook SHALL NOT modify any Vehicle_Financing record
5. WHEN the adjusted balance would fall outside the range `[0, originalAmount]`, THE Financing_Hook SHALL clamp the `currentBalance` to that range

### Requirement 6: Financing Expense Query

**User Story:** As a developer, I want to query expenses filtered by the `isFinancingPayment` flag, so that the financing tab can load only financing-related expenses for a vehicle.

#### Acceptance Criteria

1. WHEN the Financing_Tab requests financing expenses for a vehicle, THE Expense_API SHALL return only expenses where `isFinancingPayment === true` and `vehicleId` matches the requested vehicle
2. THE Expense_API SHALL return financing expenses sorted by date in ascending order

### Requirement 7: Derived Payment Entries

**User Story:** As a user, I want to see payment numbers, remaining balances, and principal/interest breakdowns on the financing tab, so that I can track my loan or lease progress.

#### Acceptance Criteria

1. WHEN financing expenses are loaded for a vehicle, THE Financing_Tab SHALL derive a sequential `paymentNumber` starting from 1 based on date-ascending sort order
2. WHEN financing expenses are loaded, THE Financing_Tab SHALL compute `remainingBalance` for each payment as `originalAmount` minus the cumulative sum of all preceding payment amounts (inclusive), clamped to a minimum of zero
3. WHEN the Vehicle_Financing is a loan with APR greater than zero, THE Financing_Tab SHALL look up `principalAmount` and `interestAmount` from the Amortization_Schedule for each payment number
4. WHEN the Vehicle_Financing is a lease or a loan without APR, THE Financing_Tab SHALL set `principalAmount` equal to the expense amount and `interestAmount` to zero
5. WHEN a payment's `expenseAmount` exceeds the Vehicle_Financing `paymentAmount`, THE Financing_Tab SHALL classify the payment as `extra`; otherwise as `standard`

### Requirement 8: Financing Checkbox in Expense Form

**User Story:** As a user, I want a checkbox in the expense form to mark an expense as a financing payment, so that I can explicitly indicate when an expense should affect my financing balance.

#### Acceptance Criteria

1. WHEN the Expense_Form category is set to `financial` AND the selected vehicle has active financing, THE Expense_Form SHALL display a checkbox labeled "Apply as payment towards financing"
2. WHEN the Expense_Form category is not `financial` OR the selected vehicle has no active financing, THE Expense_Form SHALL hide the financing payment checkbox
3. WHEN the checkbox is checked, THE Expense_Form SHALL include `isFinancingPayment: true` in the submitted payload
4. WHEN the checkbox is unchecked, THE Expense_Form SHALL include `isFinancingPayment: false` in the submitted payload

### Requirement 9: Record Payment from Financing Tab

**User Story:** As a user, I want a "Record Payment" button on the financing tab that opens the expense form pre-filled with financing data, so that recording payments is quick and convenient.

#### Acceptance Criteria

1. WHEN the Financing_Tab displays an active Vehicle_Financing, THE Financing_Tab SHALL show a "Record Payment" button
2. WHEN the user clicks "Record Payment", THE Financing_Tab SHALL navigate to the Expense_Form with `vehicleId`, `category` set to `financial`, `isFinancingPayment` set to `true`, and `amount` set to the Vehicle_Financing `paymentAmount`
3. WHEN the Vehicle_Financing is not active, THE Financing_Tab SHALL NOT display the "Record Payment" button

### Requirement 10: Financing Tab Data Source Migration

**User Story:** As a user, I want the financing tab to display payment data sourced from expenses instead of the dropped payments table, so that all my financial data is consistent and unified.

#### Acceptance Criteria

1. THE Financing_Tab SHALL load payment history by querying financing expenses from the Expense_API instead of the removed `vehicle_financing_payments` endpoint
2. THE Financing_Tab SHALL pass Derived_Payment_Entry data to the PaymentHistory, FinancingCharts, and NextPaymentCard components
3. THE Financing_Tab SHALL NOT call any API endpoint related to `vehicle_financing_payments`

### Requirement 11: API Response with Financing Data

**User Story:** As a frontend developer, I want the expense API to return updated financing data alongside the expense when a financing payment is created or modified, so that the UI can update without a separate API call.

#### Acceptance Criteria

1. WHEN a Financing_Payment expense is created, THE Expense_API SHALL include the updated Vehicle_Financing record in the response body alongside the created expense
2. WHEN a Financing_Payment expense is updated, THE Expense_API SHALL include the updated Vehicle_Financing record in the response body alongside the updated expense
3. WHEN a non-financing expense is created or updated, THE Expense_API SHALL NOT include Vehicle_Financing data in the response body
